## 1. 이 프로젝트가 무엇인가 (한 문장)

한국 주식 산업 밸류체인에서 **"금 캐는 사람(완성품 우량주)"이 아니라
"곡괭이 파는 사람(필수 공급사)"을 객관적 데이터로 찾아내는** 분석 도구.

투자업계 용어로 **picks-and-shovels 전략**. 골드러시 때 금 캐는 사람보다
청바지·곡괭이 파는 사람이 더 안정적으로 돈을 번 것처럼, 반도체·피지컬AI가
성장할 때 누가 이기든 꼭 필요한 부품·원자재·장비를 공급하며 실제로 돈을 버는
기업을 찾는다.

### 핵심 철학 (절대 잊지 말 것)
- **주가 예측기가 아니다.** "어느 종목이 얼마 갈지"는 안 한다.
  대신 "어떤 구조적 강점이 있어 더 깊이 볼 가치가 있는가"를 점수화한다.
- 결과는 매매 신호가 아니라 **후보 추리기(candidate screening)**다.
- 모든 판단은 **객관적 데이터(밸류체인 그래프 구조 + 재무지표)**에만 근거한다.
- 최종 의견은 **상반된 관점(Bull/Bear)을 대조**해서 낸다. 한쪽 주장만 신뢰하지 않는다.

## 2. 지금까지의 설계 결정 (왜 이렇게 만들었나)

### 2-1. 데이터 소스를 추상화한 이유
증권 API(토스/한국투자/pykrx)는 서로 스펙이 다르고, 토스 오픈API는 아직
키 발급 대기 중이다. 그래서 `MarketDataSource` 인터페이스를 두고,
지금은 `SampleSource`로 개발하고 나중에 구현체만 갈아끼운다.
**API에 종속되지 않고 핵심 로직을 먼저 완성하기 위한 결정.**

### 2-2. 두 개의 독립 시스템으로 분리
- **시스템 A (결정론적)**: 계좌·시세·주문 = 숫자가 틀리면 안 되는 영역. AI 개입 금지.
- **시스템 B (확률적)**: 밸류체인 분석·AI 의견 = 이 프로젝트의 진짜 가치.
현재 코드는 전부 시스템 B다. 시스템 A(실거래)는 나중에, 그것도 본인 매매용으로만.

### 2-3. 곡괭이 점수 = 4개 축의 가중합
| 축 | 의미 | 측정 방법 |
|----|------|-----------|
| irreplaceability (0.30) | 대체 불가능성 | 하류 고객의 의존도(엣지 weight) 최댓값 |
| customer_spread (0.25) | 고객 분산 | 납품 고객 수 (많을수록 "누가 이기든 돈 범") |
| upstream_position (0.15) | 상류 위치 | 하류는 있고 상류는 적을수록 공급단 |
| financial_reality (0.30) | 재무 실체 | 매출성장·영익성장·영익률·수주잔고 |

**함정 페널티**: 구조는 좋아 보여도 financial_reality < 0.2면 ×0.7.
"스토리만 있고 돈은 안 버는" 종목을 거른다.

### 2-4. 성숙도 분류 = 곡괭이 점수와 독립된 두 번째 축
곡괭이 점수(구조+실적)와 별개로 "시장이 이미 알아봤는가"를 측정해 4분면 생성:
- **핵심곡괭이** (좋고+이미 비쌈): 안전하나 상승여력 제한
- **숨은곡괭이** (좋은데+저평가): ★ 우리가 노리는 자리, 재평가 여력 큼
- **검증된성장주** (곡괭이 약함+유명): 완성품/테마주 성격
- **관심밖** (곡괭이 약함+저평가): 싼 데는 이유가 있음

시장 인식도 = PER/PBR(업종평균 대비) + 1년 주가수익률 + 시총.

### 2-5. 멀티 AI는 "경쟁"이 아니라 "역할 토론"
여러 AI를 평균내면 같이 틀린다(같은 학습데이터 편향). 그래서:
- Claude = Bull(상향 논거), Gemini = Bear(하향 논거), 제3모델 = Judge(판정)
- **모든 의견은 근거(Evidence)를 강제**한다. 근거 없는 의견은 무효(is_valid=False).
- 근거는 제공된 밸류체인+시세 데이터 안에서만 나와야 한다(환각 방지).
- 추후 백테스트로 어느 역할/모델이 맞췄는지 기록해 신뢰도 가중치 동적 조정.

## 3. 현재 코드 구조

```
valuechain/
├── CLAUDE.md             ← 이 문서
├── README.md             ← 사용자용 사용법
├── data/
│   └── source.py         MarketDataSource 인터페이스 + SampleSource(완성)
│                         + PykrxSource/TossSource(스텁, 미구현)
├── core/
│   └── graph.py          ValueChain 그래프 엔진(완성)
│                         - supplies/competes 관계, upstream/downstream
│                         - propagate_shock: 충격 전파 시뮬레이션
│                         - tech_exposure: 특정 기술 보유 기업 탐지
├── analysis/
│   ├── shovel.py         ShovelDetector 곡괭이 점수(완성)
│   ├── maturity.py       MaturityClassifier 4분면 분류(완성)
│   └── debate.py         DebateEngine 멀티AI 토론(뼈대만, LLM 호출부 미구현)
├── demo.py               충격 전파 데모 + build_semiconductor_chain()
├── demo_shovel.py        곡괭이 랭킹 데모
└── demo_full.py          전체 파이프라인 데모 (점수→분류→두 바구니 추천)
```

### 완성도 현황
- ✅ 완성·검증됨: graph.py, shovel.py, maturity.py, SampleSource, 3개 데모
- 🔲 스텁(인터페이스만): PykrxSource, TossSource, debate.py의 LLMClient
- ❌ 아직 없음: 데이터 자동수집, 산업 자동선정, 백테스트, UI

### 실행 확인
```bash
pip install pykrx finance-datareader networkx
python3 demo_full.py   # 반도체 샘플로 전체 파이프라인 동작 확인
```

## 4. 알려진 한계 (정직하게)

1. **데이터가 전부 수동 입력**이다. 공급관계와 재무가 손으로 채운 샘플.
   이게 가장 큰 약점. 자동화 전까지는 "리서치 보조 도구" 수준.
2. **공급관계(그래프 엣지) 자동수집이 가장 어렵다.** 완전 자동화 난망.
   DART 공시의 "주요 매출처/매입처"는 비정형 텍스트라 추출+검수 필요.
3. **점수 가중치가 검증 안 됨.** 백테스트로 실제 수익과의 상관을 확인하기 전엔
   임의값이다. 절대 매매 신호로 쓰면 안 된다.
4. **투자 자문이 아니다.** 면책: 이 도구의 출력은 참고용 후보일 뿐.

## 6. 작업 시 지켜야 할 원칙

- **설명하면서 진행할 것.** 사용자는 ML·자료구조를 공부하는 학생이고 코드를
  직접 이해하며 늘리는 스타일이다. "왜 이 방법인지" 설명하며 같이 간다.
- **간결하고 직접적으로.** 사용자는 군더더기 없는 설명을 선호하고, 이상하면
  바로 반문한다. 과한 칭찬·장황한 서론 금지.
- **데이터 정합성 우선.** pykrx/DART 데이터는 지저분하다(결측·이상치·종목코드
  불일치). 자동화할 때 검증·예외처리를 반드시 넣는다.
- **추상화 계층을 깨지 말 것.** 나머지 코드는 데이터 출처를 몰라야 한다.
  새 소스는 항상 MarketDataSource 인터페이스를 구현하는 방식으로 추가.
- **면책 유지.** 투자 자문이 아님을 코드 주석·출력에 명시. 매매 자동화는
  본인용·검증 후로 한정.
- **비밀키 관리.** API 키는 환경변수/.env로만. 절대 코드에 하드코딩·커밋 금지.
  .gitignore에 .env 추가.
